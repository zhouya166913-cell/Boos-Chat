# 参考项目说明

以下目录保留为迁移参考，不属于当前四项目主线：

```text
minimax-token-chat-demo/  # 旧 Node.js + SQLite 原型
MinimaxAndroidChat/       # 旧 Android 原型
tools/                    # 旧 SQLite 工具与迁移辅助工具
```

它们的用途是：

- 回看旧功能与交互
- 对照旧数据结构
- 作为后续设计时的历史样例

它们不参与当前主线项目启动，也不应继续承载新功能开发。

需要特别明确：

- 当前四个正式项目与这三个参考目录之间**没有运行依赖、构建依赖、代码依赖或数据库依赖**
- 这三个目录只是参考样例，不是四项目架构的组成部分
- `tools/` 也只服务于参考项目自身，正式四项目与 `tools/` 无关

---

## 固定参考结论

自当前阶段起，以下三个目录可视为**不会继续变化的历史参考基线**：

```text
minimax-token-chat-demo/
MinimaxAndroidChat/
tools/
```

后续如果只是为了理解旧功能、旧交互、旧数据结构或旧接口，不需要把它们当成“仍在演进中的项目”重新做全量分析；优先直接复用下面这些固定结论即可。

## 1. `minimax-token-chat-demo`

这是旧版完整原型，也是当前新架构设计时可回看的**业务参考源**。

### 固定技术事实

- 技术栈：`Node.js + SQLite + 原生 HTML/CSS/JavaScript`
- 入口：`server.js`
- 认证、数据库和大部分业务数据访问逻辑集中在：`auth.js`
- SQLite 数据库默认位于：`data/app.db`
- 默认通过 `tools/sqlite-tools-win-x64-3530000/sqlite3.exe` 访问 SQLite
- 用户端端口：`3000`
- 管理端端口：`3001`
- 用户端和管理端由同一个 Node 服务体系承载，但按端口和角色区分
- 旧认证头格式是：

```text
Authorization: Bearer <token>
```

这与新主线后端当前使用的：

```text
satoken: <token>
```

不同，迁移时不要混淆。

### 固定业务事实

旧原型已经跑通过以下主能力：

- 登录与退出
- 用户端 / 管理端分离
- 登录防爆破
- 1 个 Web 会话 + 1 个 App 会话限制
- 公司资料采集
- 文本文件上传与 AI 抽取
- 公司知识库上下文注入
- AI 对话与 SSE 流式返回
- 对话新建、归档、恢复、删除
- 个人设置
- 多模型 Provider 适配
- 用量统计
- 内容安全治理
- 管理端用户、会话、登录安全、安全标记、模型设置、额度查询

## 2. `MinimaxAndroidChat`

这是旧版 Android 原型，也是 Flutter App 后续设计时可回看的**移动端交互参考源**。

### 固定技术事实

- 技术栈：`原生 Android + Java`
- 主入口：`app/src/main/java/com/example/minimaxchat/MainActivity.java`
- 通过：

```text
http://10.0.2.2:3000
```

访问旧 Node 后端
- 旧 Android 端同样使用：

```text
Authorization: Bearer <token>
```

进行认证

### 固定业务事实

旧 Android 端已经验证过以下移动端能力：

- 登录 / 登出
- 分析对话
- 快捷分析按钮
- 公司资料采集
- 文本文件上传
- 对话归档与删除
- 个人设置
- 免责声明确认

它的价值主要在于：

- 参考移动端页面结构
- 参考老板侧使用路径
- 参考后续 Flutter App 迁移优先级

不建议把其中的 Java UI 代码当作新主线继续扩展。

## 3. `tools`

这是旧原型配套的 SQLite 工具目录，不是未来主线功能目录。

### 固定内容

当前主要包含：

```text
sqlite3.exe
sqldiff.exe
sqlite3_analyzer.exe
sqlite-tools-win-x64-3530000/sqlite3.exe
```

### 固定用途

- 支撑旧 `minimax-token-chat-demo` 自身读取 SQLite 数据库
- 查询旧库表结构和历史数据
- 供需要回看参考项目时使用

### 不应误解为

- 不是当前新后端的数据库运行依赖
- 不承担未来业务功能开发
- 不应因为它还存在，就把主线继续往 SQLite 上扩展

## 后续接手时的使用原则

1. 先把三个参考目录当成**历史参考样例**，不是活跃开发对象。  
2. 需要了解旧功能时，优先查本文件的固定结论，再按需回看旧代码。  
3. 新需求、新接口、新数据结构，仍然优先落在四个主线项目中。  
4. 需要做方案设计时，可按需借鉴：
   - `minimax-token-chat-demo` 的业务能力与数据模型
   - `MinimaxAndroidChat` 的移动端交互路径
   - `tools` 仅在查看参考项目时按需使用
