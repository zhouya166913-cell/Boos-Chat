# 企业管理助手 — 前后端分离整改文档

> 基于现有 Node.js + SQLite + 原生 HTML 项目，重构为 Java Spring Boot + MySQL 后端、Vue 3 管理系统、Flutter App、uni-app 小程序四项目架构。

## 当前落地状态说明

本文档主要保留为**整体改造方案与目标设计文档**，其中有些内容描述的是后续目标态，不等同于当前已经落地的实现。

截至当前版本，已经落地的实际状态为：

- 接口文档当前使用 `springdoc-openapi`，Swagger 入口为 `http://localhost:9090/swagger-ui.html`
- 当前正式登录接口为：

```text
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
```

- 当前已落地第一版正式认证表：

```text
sys_user
sys_role
sys_permission
sys_user_role
sys_role_permission
sys_oauth_account
sys_login_record
```

- Redis、完整在线会话管理、登录防爆破等仍属于后续目标能力，尚未全部落地

因此：

- **查看当前可用接口时，以 Swagger 和当前代码为准**
- **查看长期演进方向时，再参考本文档中的目标方案**

## 四、整改注意事项

### 4.1 架构层面

| 序号 | 注意事项 | 说明 |
|------|---------|------|
| 1 | **前后端完全分离部署** | 原项目 Node.js 同时服务静态文件和 API，新架构前端独立部署（Nginx 托管），后端只提供 REST API |
| 2 | **CORS 跨域配置** | 前后端分离后，后端必须配置 CORS 允许前端域名访问；生产环境通过 Nginx 反向代理同域可避免 |
| 3 | **SSE 流式接口兼容** | 原项目 `/api/chat` 使用 SSE 流式输出，Spring Boot 需用 `SseEmitter` 或 `WebFlux Flux<ServerSentEvent>` 实现，前端 Axios 不支持 SSE，需用 `EventSource` 或 `fetch` + `ReadableStream` |
| 4 | **双端口变单端口** | 原项目用户端 3000 + 管理端 3001 双端口，新架构统一为单后端端口，通过路由前缀 `/api/user/*` 和 `/api/admin/*` 区分，前端通过不同 SPA 入口或路由守卫区分 |
| 5 | **会话管理迁移** | 原项目自研 Bearer Token + SQLite sessions 表，新架构用 Sa-Token + Redis，需保持同一账号 Web/App 会话数量限制的业务逻辑 |
| 6 | **SQLite → MySQL 数据迁移** | 如需保留旧数据，需编写迁移脚本将 SQLite 数据导入 MySQL |

### 4.2 业务逻辑层面

| 序号 | 注意事项 | 说明 |
|------|---------|------|
| 1 | **登录防爆破策略必须保留** | 原项目有完善的防爆破体系（账号连续失败锁定、日失败锁定、IP 窗口限制、管理员特殊策略），Sa-Token 内置部分能力，但 IP 级别限制和日失败锁定需要自行扩展 |
| 2 | **同一账号会话数量限制** | 原项目限制同一账号 1 Web + 1 App 会话，Sa-Token 需通过监听器自定义实现 |
| 3 | **内容安全治理必须保留** | 政治类拦截和违规脱敏是核心合规要求，迁移时正则规则和记录逻辑必须完整保留 |
| 4 | **LLM Provider 适配层需重新设计** | 原项目用 `fetch` 直接调用各模型商 API，Java 端需用 `WebClient` 或 `OkHttp` 实现，注意不同 Provider 的参数差异（`max_completion_tokens` vs `max_tokens`、reasoning 模型去掉 temperature 等） |
| 5 | **公司知识库上下文注入** | 原项目在 `/api/chat` 中自动将公司资料拼接到 system message，这个逻辑必须在新后端完整复刻 |
| 6 | **免责声明只弹一次** | 原项目通过 `user_settings.data_disclaimer_accepted_at` 字段控制，新架构需保留此逻辑 |
| 7 | **用户端不暴露技术信息** | 原项目用户端只显示"AI 调用次数"，不显示模型名、Token、Provider 等，新架构 API 返回值需保持同样的信息隔离 |
| 8 | **OpenAI 联网搜索** | 使用的是 Responses API（非 Chat Completions），请求格式和返回格式与普通对话完全不同，需单独实现 |
| 9 | **文件上传大小限制** | 原项目限制请求体 2MB（资料保存）和 5MB（文件分析），Spring Boot 需配置 `spring.servlet.multipart.max-file-size` |
| 10 | **Token 估算方式** | 原项目用 `字符数 / 4` 粗估，新架构可保持一致或引入 tiktoken 精确计算 |

### 4.3 前端迁移层面

| 序号 | 注意事项 | 说明 |
|------|---------|------|
| 1 | **三套界面拆分为独立路由** | 原项目登录页 + 用户端 + 管理端在同一个 HTML 中通过 `hidden` 切换，新架构应拆分为独立路由和布局组件 |
| 2 | **SSE 流式读取方式变化** | 原项目用 `fetch` + `ReadableStream`，Vue 3 中需封装为 composable 或使用 `EventSource` polyfill |
| 3 | **状态管理需重新设计** | 原项目用全局 `state` 对象，新架构用 Pinia store 拆分为 `useAuthStore`、`useChatStore`、`useProfileStore`、`useAdminStore` 等 |
| 4 | **响应式布局用 Element Plus 栅格** | 原项目手写大量 CSS 媒体查询，Element Plus 的 `el-row` / `el-col` + `xs/sm/md/lg` 可大幅简化 |
| 5 | **管理端表格用 Element Plus Table** | 原项目管理端表格是手写 HTML 拼接，Element Plus 的 `el-table` + `el-pagination` + `el-form` 可大幅简化 |
| 6 | **免责声明弹窗用 el-dialog** | 原项目手写 modal，迁移为 `el-dialog` 组件 |

### 4.4 原 Android → Flutter App 迁移层面

| 序号 | 注意事项 | 说明 |
|------|---------|------|
| 1 | **Java 代码 UI → Widget 树** | 原 Android 端全部用 Java 代码构建 UI，Flutter 需要按页面、组件重新拆分 Widget |
| 2 | **HttpURLConnection → Dart 网络层** | 需统一封装请求、Token 注入、错误处理和超时策略 |
| 3 | **SSE 流式需单独设计** | Flutter 端要结合实际 Provider 选择流式 HTTP、WebSocket 或其他稳定方案 |
| 4 | **文件选择需引入插件** | Android 原生文件选择逻辑要迁移为 Flutter 对应能力，并提前考虑 iOS 权限差异 |
| 5 | **本地存储能力重建** | 原 `SharedPreferences` 需要迁移到 Flutter 侧的安全存储或键值存储方案 |
| 6 | **鸿蒙后续单独适配** | 当前 App 主线先覆盖 Android / iOS，鸿蒙不在本阶段强行兼容 |

### 4.5 小程序开发层面

| 序号 | 注意事项 | 说明 |
|------|---------|------|
| 1 | **小程序独立成项目** | 小程序不与 App 共用工程，仅共用后端接口和业务约定 |
| 2 | **网络请求统一封装** | 使用 `uni.request` 封装 Token、错误码和登录失效处理 |
| 3 | **正式环境必须 HTTPS** | 微信和支付宝小程序都需要按平台要求配置合法请求域名 |
| 4 | **平台能力存在差异** | 微信和支付宝在登录、授权、上传、审核规则上不能完全视为一致 |
| 5 | **H5 预览不等于小程序运行时** | `9900` 只是本地 H5 调试端口，正式小程序并不依赖该端口 |

## 五、MySQL 数据库设计方案

### 5.1 设计原则

1. **保持与原 SQLite 表结构的功能对等**，确保业务逻辑完整迁移
2. **字段命名采用 snake_case**，符合 MySQL 惯例
3. **所有表增加 `is_deleted` 逻辑删除字段**，替代原项目的物理删除
4. **所有表增加 `create_time` / `update_time`**，由 MyBatis-Plus 自动填充
5. **使用 InnoDB 引擎**，支持事务和外键
6. **字符集 utf8mb4**，支持中文和 emoji
7. **敏感字段（密码哈希、Token 哈希）不可逆加密存储**

### 5.2 完整表结构

#### 5.2.1 用户表 `sys_user`

```sql
CREATE TABLE sys_user (
    id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '用户ID',
    username        VARCHAR(40)     NOT NULL COMMENT '登录账号',
    password_hash   VARCHAR(200)    NOT NULL COMMENT '密码哈希(BCrypt)',
    display_name    VARCHAR(80)     NOT NULL DEFAULT '' COMMENT '显示名',
    role            VARCHAR(20)     NOT NULL DEFAULT 'user' COMMENT '角色：admin/user',
    status          TINYINT         NOT NULL DEFAULT 1 COMMENT '状态：1正常 0禁用',
    is_deleted      TINYINT         NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除 1已删除',
    create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

**与原表对比**：
- `password_hash` 从 scrypt 改为 BCrypt（Spring Security Crypto 原生支持）
- 新增 `status` 字段，支持账号禁用（原项目无此功能，属于增强）
- 新增 `is_deleted` 逻辑删除

#### 5.2.2 会话表 `sys_session`

```sql
CREATE TABLE sys_session (
    id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '会话ID',
    user_id         BIGINT          NOT NULL COMMENT '用户ID',
    token_hash      VARCHAR(64)     NOT NULL COMMENT 'Token SHA-256 哈希',
    client_type     VARCHAR(20)     NOT NULL DEFAULT 'web' COMMENT '客户端类型：web/android',
    client_name     VARCHAR(120)    NOT NULL DEFAULT '' COMMENT '客户端标识',
    ip_address      VARCHAR(120)    NOT NULL DEFAULT '' COMMENT '登录IP',
    user_agent      VARCHAR(500)    NOT NULL DEFAULT '' COMMENT '浏览器UA',
    expires_at      DATETIME        NOT NULL COMMENT '过期时间',
    last_seen_at    DATETIME        DEFAULT NULL COMMENT '最后活跃时间',
    revoked_at      DATETIME        DEFAULT NULL COMMENT '撤销时间',
    revoked_reason  VARCHAR(40)     NOT NULL DEFAULT '' COMMENT '撤销原因',
    revoked_by      BIGINT          DEFAULT NULL COMMENT '撤销操作人ID',
    create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_token_hash (token_hash),
    KEY idx_user_client (user_id, client_type, revoked_at, expires_at),
    KEY idx_last_seen (last_seen_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='会话表';
```

**设计思考**：
- Sa-Token 默认用 Redis 存储会话，但为了管理端能查看在线会话列表、强制下线等功能，**同时在 MySQL 中保留会话记录**
- Redis 作为一级缓存（快速验证 Token），MySQL 作为二级存储（管理端查询）
- `last_seen_at` 用于判断会话是否在线，需配合心跳接口更新

#### 5.2.5 登录锁定表 `sys_login_lock`

```sql
CREATE TABLE sys_login_lock (
    id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '锁定ID',
    user_id         BIGINT          DEFAULT NULL COMMENT '用户ID',
    username        VARCHAR(40)     NOT NULL DEFAULT '' COMMENT '被锁定账号',
    ip_address      VARCHAR(120)    NOT NULL DEFAULT '' COMMENT '被锁定IP',
    scope           VARCHAR(20)     NOT NULL COMMENT '锁定范围：account/ip',
    reason          VARCHAR(80)     NOT NULL DEFAULT '' COMMENT '锁定原因',
    locked_until    DATETIME        NOT NULL COMMENT '锁定到期时间',
    unlocked_at     DATETIME        DEFAULT NULL COMMENT '解锁时间',
    unlocked_by     BIGINT          DEFAULT NULL COMMENT '解锁操作人ID',
    create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    PRIMARY KEY (id),
    KEY idx_active_lock (scope, username, ip_address, locked_until, unlocked_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='登录锁定表';
```

**设计思考**：
- 原项目的防爆破逻辑在 Java 中用 Redis + MySQL 双写实现
- Redis 用于实时计数（快速判断是否触发锁定），MySQL 用于持久化记录和日志导出
- 锁定检查优先查 Redis，MySQL 作为兜底

    company_intro   TEXT            NOT NULL COMMENT '公司介绍',
    contact_address TEXT            NOT NULL COMMENT '联系地址',
    contact_methods TEXT            NOT NULL COMMENT '联系方式',
    main_business   TEXT            NOT NULL COMMENT '主营业务',

### 5.5 Redis 缓存设计

| Key 模式 | 数据类型 | 用途 | TTL |
|----------|---------|------|-----|
| `session:token:{hash}` | Hash | Token → 用户信息快速查询 | 7 天 |
| `session:online:{userId}:{clientType}` | String | 在线会话标记 | 5 分钟（心跳续期） |
| `login:attempt:account:{username}` | String (计数器) | 账号连续失败次数 | 30 分钟 |
| `login:attempt:account:daily:{username}` | String (计数器) | 账号日失败次数 | 24 小时 |
| `login:attempt:ip:{ip}` | String (计数器) | IP 窗口内失败次数 | 10 分钟 |
| `login:lock:account:{username}` | String | 账号锁定标记 | 锁定时长 |
| `login:lock:ip:{ip}` | String | IP 锁定标记 | 锁定时长 |
| `setting:app:{key}` | String | 应用配置缓存 | 1 小时 |
| `usage:websearch:daily:{userId}` | String (计数器) | 当日联网搜索次数 | 当天结束 |

## 八、开发优先级建议

### Phase 1：基础骨架

1. 后端 Spring Boot 项目搭建 + MySQL 建表 + MyBatis-Plus 代码生成
2. Sa-Token 登录认证 + Redis 会话管理
3. 前端 Vue 3 + Element Plus 项目搭建 + 登录页
4. 前后端联调登录流程

### Phase 2：核心业务

5. LLM Provider 适配层（先实现 MiniMax 一个 Provider）
6. 对话功能（SSE 流式）+ 对话管理
7. 公司资料采集 + 文件上传分析
8. 公司分析快捷入口

### Phase 3：管理功能

9. 管理端用户管理
10. 在线会话管理
11. 登录安全 + 防爆破
12. 内容安全治理
13. 模型设置 + 额度查询

### Phase 4：多端客户端

14. Flutter App 项目完善 + 登录联调
15. uni-app 小程序项目完善 + 登录联调
16. App 与小程序逐步迁移对话、资料采集、个人设置等业务能力
17. 根据真实需求继续做平台专项适配
