# 项目代码规范说明

本文档用于统一当前四项目架构下的代码风格、目录职责、注释要求和协作约定。

关于“目录如何长期组织、页面/逻辑/样式什么时候需要拆分”，统一以《项目目录与文件组织规范》为准；本文档更侧重命名、注释、接口与协作规则。

适用范围：

```text
boss-chat-server/     # Java Spring Boot 后端
boss-chat-web/        # Vue 3 + TypeScript + Element Plus 管理系统
boss-chat-app/        # Flutter App
boss-chat-miniapp/    # Vue 3 + uni-app 小程序
```

目标不是“让代码看起来工整”，而是让任何一个后来接手的人都能尽快回答下面几个问题：

1. 这段代码负责什么？
2. 它依赖了什么接口或数据？
3. 入口在哪里？
4. 关键方法为什么这样写？
5. 修改它会影响哪里？

---

## 一、总体原则

### 1. 先清晰，再简洁

- 优先写让人容易理解的代码
- 不为了少几行代码牺牲可读性
- 能一眼看懂的地方不必过度抽象
- 业务逻辑一旦出现明显分支、状态切换或隐含规则，就要适当拆分并补充说明

### 2. 一个文件只承担一类清晰职责

- Controller 不写复杂业务逻辑
- Service 不直接承载页面展示逻辑
- Mapper 只处理数据访问
- 页面组件不把接口请求、状态管理、展示逻辑全部揉成一团

### 3. 当前项目要按真实长期项目维护

- 不写只为“临时跑通”的命名
- 不保留无意义测试代码
- 不把临时实现伪装成正式实现
- 未来可能扩展的地方，要通过清晰命名和适度注释留下边界

### 4. 注释的目标是补充“代码本身看不出来的信息”

好的注释回答：

- 为什么这样做
- 这个接口是给谁用的
- 这个组件依赖哪些接口
- 这个方法处理什么业务场景
- 哪些边界或前提不能随便改

不好的注释只是把代码翻译成中文，例如：

```java
// 设置用户名
user.setUsername(username);
```

这类注释通常没有价值。

---

## 二、命名规范

### 1. 通用命名

- 类、组件、页面名要体现职责
- 变量名要体现真实含义，避免 `data`、`list`、`temp`、`info` 这种过宽命名
- 布尔值统一使用：
  - `isXxx`
  - `hasXxx`
  - `canXxx`
- 时间字段优先使用：
  - `createTime`
  - `updateTime`
  - `lastLoginTime`
- ID 字段统一：
  - `userId`
  - `roleId`
  - `permissionId`

### 2. 后端命名

| 类型 | 约定 |
| --- | --- |
| Controller | `XxxController` |
| Service | `XxxService` |
| DTO | `XxxRequest` / `XxxResponse` |
| Entity | 与表语义一致，如 `SysUser` |
| Mapper | `XxxMapper` |
| 配置类 | `XxxConfig` |

### 3. 前端命名

| 类型 | 约定 |
| --- | --- |
| 页面组件 | `XxxView.vue` |
| 布局组件 | `XxxLayout.vue` |
| 通用组件 | 语义化 PascalCase，如 `UserStatusTag.vue` |
| API 文件 | 以业务域命名，如 `auth.ts`、`user.ts` |
| Store | 以业务域命名，如 `auth.ts` |
| 事件处理方法 | `handleXxx` |
| 数据加载方法 | `loadXxx` / `fetchXxx` |

### 4. Flutter / uni-app 命名

- Flutter Widget 类使用 PascalCase
- Dart 私有字段与私有方法使用 `_xxx`
- uni-app 页面、模块、接口文件沿用前端命名规则
- 跨端共用概念尽量保持命名一致，例如：
  - `CurrentUser`
  - `LoginResponse`
  - `AuthApi`

## 三、目录职责规范

### 1. 后端 `boss-chat-server`

当前推荐目录：

```text
auth/       # 登录、认证、会话相关
system/     # 用户、角色、权限等系统基础能力
common/     # 公共配置、异常、通用能力
```

约定：

- 新业务模块按业务域拆包，不把所有代码堆到 `common`
- Controller、DTO、Service、Mapper、Entity 要分层清晰
- 表结构变更必须对应 Flyway 迁移脚本
- 不允许只改实体类、不改数据库迁移

### 2. 管理系统 `boss-chat-web`

推荐目录：

```text
api/        # 接口请求封装
layouts/    # 页面布局
router/     # 路由
stores/     # 状态管理
views/      # 页面级组件
components/ # 可复用业务组件，后续按需新增
```

约定：

- 页面级代码放 `views`
- 可复用组件不要塞在页面里反复复制
- API 请求统一经由 `api` 层封装
- Store 只处理跨页面共享状态，不把所有局部状态都塞进 Store

### 3. Flutter App `boss-chat-app`

当前项目还处于起步阶段，后续功能增加后推荐逐步拆为：

```text
lib/
├─ api/          # 接口访问
├─ models/       # 数据模型
├─ pages/        # 页面
├─ widgets/      # 可复用组件
├─ services/     # 业务服务
└─ main.dart
```

约定：

- 页面、接口、模型不要长期堆在 `main.dart`
- 当某个页面超过一个清晰职责时，应及时拆文件

### 4. 小程序 `boss-chat-miniapp`

当前推荐目录：

```text
api/        # 接口请求
config/     # 环境配置
pages/      # 页面
types/      # 类型定义
utils/      # 工具方法
components/ # 后续通用组件
```

约定：

- 页面里只保留与页面强相关的状态和交互
- 请求逻辑统一放在 `api`
- 通用请求包装统一走 `utils/request.ts`

## 四、注释规范

这是本项目的强制要求。

### 1. 后端接口必须写清楚说明

所有对外 Controller 接口必须至少说明：

1. 接口作用
2. 使用场景
3. 关键入参含义
4. 返回结果含义
5. 需要权限时，说明权限要求

当前项目已有 Swagger / OpenAPI，接口至少应具备：

- `@Tag`
- `@Operation`
- 必要时为参数、字段补充 Schema 说明

对于关键接口，建议额外补充 JavaDoc。

示例：

```java
/**
 * 使用本地账号密码登录系统。
 *
 * <p>登录成功后返回当前会话 token 与用户基础信息，
 * 供管理系统、App、小程序统一使用。</p>
 */
@Operation(summary = "账号密码登录")
@PostMapping("/login")
public LoginResponse login(@Valid @RequestBody LoginRequest request) {
    return authService.login(request);
}
```

要求：

- 接口注释不能只写“登录”“查询列表”这种过短描述
- 如果接口有业务限制，例如“仅超级管理员可用”，必须写明
- DTO 字段如果含义不直观，要写字段说明

### 2. 后端业务方法需要说明“做什么”和“为什么”

以下方法必须补充注释：

- 复杂业务逻辑
- 涉及权限判断的方法
- 涉及状态流转的方法
- 涉及第三方平台交互的方法
- 与数据库结构存在隐含约束的方法

示例：

```java
/**
 * 组装当前用户的前端可见身份信息。
 *
 * <p>权限始终从本地用户角色关系中解析，
 * 与用户本次使用账号密码、微信还是 QQ 登录无关。</p>
 */
private CurrentUserResponse toCurrentUser(SysUser user) {
    ...
}
```

### 3. 前端页面 / 模块 / 组件必须写顶部说明

所有页面级组件、复用组件、接口模块，都应在文件顶部写明：

1. 当前模块负责什么
2. 依赖了哪些接口
3. 它在页面链路中的角色

Vue 示例：

```ts
/**
 * 登录页。
 *
 * 依赖接口：
 * - POST /api/auth/login：提交账号密码并获取登录 token
 *
 * 页面职责：
 * - 采集账号密码
 * - 调用登录接口
 * - 登录成功后跳转到管理系统首页
 */
```

接口模块示例：

```ts
/**
 * 认证接口模块。
 *
 * 对应后端接口：
 * - POST /api/auth/login
 * - GET /api/auth/me
 * - POST /api/auth/logout
 */
```

### 4. 前端方法必须说明方法作用

以下方法必须写注释：

- 调用接口的方法
- 提交、保存、删除、重置等会改变状态的方法
- 页面初始化方法
- 处理复杂联动、权限或异常分支的方法

示例：

```ts
/**
 * 提交登录表单。
 *
 * 作用：
 * 1. 校验账号密码
 * 2. 调用登录接口
 * 3. 保存 token
 * 4. 跳转到仪表盘
 */
async function handleSubmit() {
  ...
}
```

### 5. 前端组件必须说明“输入、输出、依赖”

对于复用组件，注释要尽量说明：

- `props` 表示什么
- 会触发哪些 `emit`
- 是否依赖接口
- 适用于什么场景

示例：

```ts
/**
 * 用户状态标签组件。
 *
 * 输入：
 * - status：用户启用状态
 *
 * 输出：
 * - 无
 *
 * 说明：
 * - 只负责展示，不负责请求和状态修改
 */
```

### 6. Flutter App 同样要有模块与方法说明

对于 Flutter：

- 页面类要说明页面职责
- API 类要说明对应后端接口
- 复杂方法要说明业务流程

示例：

```dart
/// 认证接口客户端。
///
/// 对应后端接口：
/// - POST /api/auth/login
/// - GET /api/auth/me
/// - POST /api/auth/logout
class AuthApi {
  ...
}
```

```dart
/// 提交账号密码并尝试登录。
///
/// 成功时把 token 和用户信息回传给上层，
/// 失败时展示后端返回的错误信息。
Future<void> _submit() async {
  ...
}
```

### 7. uni-app 小程序同样执行相同注释规则

页面文件建议说明：

- 页面用途
- 依赖接口
- 主要交互

示例：

```ts
/**
 * 小程序登录页。
 *
 * 依赖接口：
 * - POST /api/auth/login
 *
 * 主要职责：
 * - 提交账号密码
 * - 持久化 token
 * - 登录成功后跳转首页
 */
```

## 五、接口与数据规范

### 1. 前后端接口路径要统一

- 后端真实接口以当前代码和 Swagger 为准
- 业务接口统一走 `/api/**`
- 前端 API 文件不得硬编码散落请求地址

### 2. 前端调用接口必须集中管理

禁止在页面中到处直接写：

```ts
axios.get(...)
```

应统一封装在：

```text
api/
```

中，再由页面调用。

### 3. 类型要同步

- 后端新增 DTO 后，前端要同步维护对应类型
- 同一概念在不同端尽量保持字段名一致
- 如果接口字段调整，必须同时检查：
  - 管理系统
  - App
  - 小程序

---

## 六、数据库与迁移规范

### 1. 数据库结构变更必须通过 Flyway

- 禁止依赖 ORM 自动改表
- 禁止只改实体类，不补迁移脚本
- 每个迁移脚本只做一类清晰变更

### 2. 已执行脚本不随意修改

- 新变更通过新版本脚本继续向前推进
- 如涉及删表、删字段、改类型，必须先评估迁移和回滚风险

### 3. 表与字段命名

- 表名使用小写下划线
- 关系表命名体现双方关系，例如：
  - `sys_user_role`
  - `sys_role_permission`
- 字段名语义明确，避免模糊缩写

---

## 七、异常与日志规范

### 1. 后端

- 不向前端暴露无意义堆栈信息
- 业务异常要给出可理解的信息
- 安全相关日志、登录日志要有明确原因码

### 2. 前端

- 用户可感知错误要给出友好提示
- 开发调试信息不要直接裸露给用户
- 与接口相关的异常分支要在方法注释或代码中体现

---

## 八、提交前自查

每次新增或修改代码前，至少自查：

1. 目录是否放对
2. 命名是否能看懂
3. 新接口是否有说明
4. 新组件 / 页面是否有顶部说明
5. 调用接口的方法是否写了作用说明
6. 是否出现重复逻辑可以抽离
7. 数据库变更是否有 Flyway 脚本
8. 是否影响管理系统、App、小程序中的同名模型
9. 是否补了必要测试或至少完成了手工验证

---

## 九、项目内统一约定

### 必须做

- 后端接口必须有清晰注释与 Swagger 说明
- 前端页面、模块、组件必须说明职责和依赖接口
- 前端关键方法必须说明作用
- 跨端共用接口发生变化时，要同步检查三端
- 涉及认证、权限、第三方登录的代码，必须写明边界与规则

### 不建议做

- 为每一行普通代码写无意义注释
- 把临时实现写成长期方案
- 在页面中散落接口调用
- 用过度抽象掩盖简单逻辑
- 不更新文档就修改关键约定

---

## 十、后续执行建议

从现在开始，后续新增代码建议统一遵守：

1. **新增接口先写清用途**
2. **新增页面先写清职责**
3. **新增组件先写清输入 / 输出 / 是否依赖接口**
4. **新增关键方法先写清作用**
5. **代码改完后同步补文档**

这样做会让当前项目即使逐步扩成后端 + 管理系统 + App + 小程序四条线，也仍然保持可维护。
