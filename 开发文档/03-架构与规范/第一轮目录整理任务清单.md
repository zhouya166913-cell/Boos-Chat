# 第一轮目录整理任务清单

本文档记录当前四个正式项目的第一轮整理范围。目标不是一次性把所有历史代码全部重写，而是先把**方向、样板和最该先收束的地方**固定下来。

---

## 一、优先级判断

| 优先级 | 项目 | 原因 |
| --- | --- | --- |
| P0 | `boss-chat-web` | 当前功能最完整，已经出现页面、状态、样式逐渐混杂的趋势，最适合先做成样板 |
| P1 | `boss-chat-server` | 业务结构基本正确，但仍有正式项目中不该长期保留的示例包 |
| P2 | `boss-chat-app` | 当前只有 `main.dart`，现在不整理，后续功能一多返工成本会更高 |
| P3 | `boss-chat-miniapp` | 结构相对清楚，可等恢复功能开发前再按新规范迁移 |

---

## 二、第一轮具体任务

## 1. `boss-chat-web`

### 本轮必须完成

- [x] 建立模块化前端目录方向
- [x] 拆分智能体对话模块
  - [x] 页面入口保留在 `views/admin/ChatView.vue`
  - [x] 状态与接口编排迁到 `modules/chat/composables/`
  - [x] 展示区块迁到 `modules/chat/components/`
  - [x] 聊天专属样式迁到 `modules/chat/styles/`
- [x] 整理登录页
  - [x] 统一格式
  - [x] 将表单逻辑迁到 `modules/auth/composables/`
  - [x] 将登录页样式迁到 `modules/auth/styles/`
- [x] 拆分全局样式
  - [x] `base.css`
  - [x] `layout.css`
  - [x] `pages.css`
  - [x] `main.css` 仅做聚合导入

### 本轮暂不强制

- 用户管理页、智能体管理页先不做深拆
- API 文件本轮保留在 `src/api/`，后续随着模块继续增长再决定是否迁入各业务模块内部
- 先不引入过多额外工具链，避免现在把精力放在“配规则”而不是“让代码结构先变清楚”

---

- [ ] 后续恢复 App 开发时，所有新功能直接按模块落位

## 4. `boss-chat-miniapp`

### 下一轮建议完成

- [ ] 增加 `modules/` 与 `shared/` 方向

- [ ] 后续新页面不再长期只堆在 `pages/`

---

## 三、验收标准

第一轮整理完成后，应满足：

1. 新同事能通过目录快速判断代码放在哪里
2. `boss-chat-web` 中至少有一个完整模块示范
3. 页面、逻辑、样式之间的职责已经开始分开
4. 后续继续整理另外三个项目时，有明确可复制的模板
5. 文档和真实代码不互相打架

---

## 四、执行原则

- 不做脱离当前开发节奏的大范围返工
- 优先整理“已经开始变复杂”的模块
- 每次触碰历史代码时，顺手把它往目标结构靠近一点
- 先把最容易成为未来混乱源头的地方处理掉
