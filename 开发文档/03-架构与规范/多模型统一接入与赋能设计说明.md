# 多模型统一接入与赋能设计说明

本文档记录当前项目关于“大模型、AI 助手、智能体、工具赋能”的最新产品与技术理解。

这份文档的目的，是避免后续继续纠结“某一家厂商提供的是大模型还是智能体”。在本项目中，外部厂商主要提供模型能力；真正的智能体能力，由我们的系统统一接入、统一配置、统一赋能。

---

## 一、核心结论

当前项目的长期方向应理解为：

```text
大模型 = 大脑
工具 = 手
记忆 = 经验
知识库 = 资料
工作流 = 做事方法
权限 = 边界
智能体 = 大脑 + 手 + 经验 + 资料 + 做事方法 + 边界
```

因此：

- 不必纠结某个外部平台宣传自己是“大模型”还是“智能体”
- 市面上主流模型都可以作为“大脑”接入系统
- 系统通过后台配置给模型赋能
- 只会回答问题的是基础模型助手
- 具备知识、记忆、流程的是增强型助手
- 能调用工具执行任务的是工具型智能体

---

## 二、概念分层

### 1. 模型供应商

模型供应商是“大脑来源”。

例如：

```text
智谱
Kimi / Moonshot
通义千问
DeepSeek
豆包
MiniMax
OpenAI
Claude
Gemini
```

供应商层主要关注：

- `provider_code`
- `provider_name`
- `base_url`
- `auth_type`
- `enabled`

注意：第二版开始，`API Key` 不再直接作为供应商级配置理解。供应商只负责记录厂家入口和认证方式；真正可调用的 Key 绑定到具体模型上。

### 2. 模型

模型是具体可调用的大脑。

例如：

```text
glm-4-flash
glm-4.7
moonshot / kimi 相关模型
deepseek-chat
qwen-plus
doubao-pro
gpt 系列
claude 系列
gemini 系列
```

模型层主要关注：

- 属于哪个供应商
- 模型名称
- 该模型可使用的 API Key
- 上下文长度
- 是否支持流式输出
- 是否支持工具调用
- 是否支持视觉 / 文件 / 多模态
- 是否启用

### 3. AI 助手

AI 助手是我们系统中的业务配置对象。

它不是某一家厂商直接给的东西，而是我们自己创建的：

```text
AI 助手 = 模型 + 系统提示词 + 能力开关 + 权限边界
```

例如：

```text
AI获客操盘手
企业AI赋能顾问
企业AI落地执行助手
客户诊断助手
合同分析助手
财务分析助手
```

### 4. 赋能能力

系统可以给 AI 助手开启不同能力：

```text
长期记忆
知识库
工作流
工具调用
文件读取
文件修改
命令执行
Git
测试
外部系统接口
```

这些能力决定 AI 助手最终表现出来的产品形态。

---

## 三、能力状态定义

后续不建议只用“大模型 / 智能体”二选一来描述 AI。

建议按能力组合展示：

| 类型 | 判断逻辑 | 含义 |
| --- | --- | --- |
| 基础模型助手 | 未开启记忆、知识库、工作流、工具 | 只负责文本问答、分析、生成 |
| 增强型助手 | 开启记忆 / 知识库 / 工作流，但未开启工具 | 可以结合资料、经验和流程回答 |
| 工具型智能体 | 开启工具调用 | 可以调用工具完成任务 |
| 本地工作区智能体 | 开启工具调用，并具备本地文件 / 命令能力 | 可以读写文件、执行命令、Git、测试 |

当前项目第一阶段可以先落地前三类：

```text
基础模型助手
增强型助手
工具型智能体
```

---

## 四、对现有字段的重新理解

当前 `ai_agent` 中已经有这些字段：

```text
memory_enabled
knowledge_enabled
workflow_enabled
tools_enabled
```

新的理解是：

| 字段 | 含义 |
| --- | --- |
| `memory_enabled` | 是否启用长期记忆 |
| `knowledge_enabled` | 是否启用知识库 |
| `workflow_enabled` | 是否启用工作流 |
| `tools_enabled` | 是否启用工具调用 |

注意：

- `tools_enabled = 1` 时，可以认为它具备工具型智能体能力。
- `tools_enabled = 0` 时，并不代表它只能是纯大模型；它仍然可以通过记忆、知识库、工作流成为增强型助手。
- 因此后续不应把 `memory_enabled / knowledge_enabled / workflow_enabled` 强行绑定到 `tools_enabled`。

## 五、后台管理建议

后续管理系统可以逐步拆成以下模块：

### 1. 模型供应商管理

用于管理 Kimi、智谱、DeepSeek、通义等模型供应商。

建议字段：

```text
provider_code
provider_name
base_url
auth_type
enabled
remark
```

### 2. 模型管理

用于管理每个供应商下的具体模型。

建议字段：

```text
provider_id
model_name
display_name
context_window
supports_stream
supports_tools
supports_vision
enabled
remark
```

### 3. AI 助手管理

当前已有 `ai_agent`，后续可以继续演进。

建议长期字段：

```text
agent_code
agent_name
description
model_id
system_prompt
temperature
max_completion_tokens
memory_enabled
knowledge_enabled
workflow_enabled
tools_enabled
enabled
```

### 4. 工具管理

当前工具是代码内置，后续可以做成后台可维护。

建议字段：

```text
tool_code
tool_name
risk_level
requires_approval
enabled
description
```

## 六、对老板或非技术人员的解释

可以这样解释：

> 市面上的大模型可以理解成 AI 的大脑，比如智谱、Kimi、DeepSeek、通义。  
> 我们系统做的不是简单接某一家模型，而是把不同模型统一接入，然后在后台给它们配置知识库、记忆、工作流和工具。  
> 当一个模型只回答问题时，它是普通 AI 助手；当我们给它资料和流程，它就是增强助手；当我们再给它工具和权限，它就能像智能体一样执行任务。

---

## 七、当前阶段落地原则

当前阶段先不要一次性大改数据库。

建议步骤：

1. 先用配置方式接入第二家模型供应商，例如 Kimi
2. 在现有 `ai_agent` 中继续复用 `model_provider / model_name`
3. 先验证多模型调用链路是否跑通
4. 再设计 `ai_model_provider / ai_model` 等正式表
5. 最后再把管理系统升级为多供应商、多模型统一管理

这样可以避免过早设计太重，同时保持长期方向正确。

---

## 八、当前结论

本项目真正要做的不是“调用某一家智能体”，而是：

```text
统一接入多个大模型
统一管理模型配置
统一配置提示词
统一绑定知识库
统一绑定工作流
统一授权工具
统一记录会话
统一审计调用
统一控制权限
```

这才是企业 AI 赋能平台的基础形态。

## 九、2026-05-19 第二版落地口径

基于第一版试用后的收敛，第二版不再把“模型管理、智能体管理、智能体工作台”拆得过细，而是采用更清晰的后台管理逻辑：

```text
模型管理：管理供应商、模型、API Key 池
智能体管理：选择模型，并给模型配置提示词、记忆、知识库、工作流、本地工具等能力
AI 对话：统一体验入口，普通大模型和工具型智能体都从这里使用
```

2026-05-19 二次修正后的关系为：

```text
供应商
  └─ 模型
      └─ 模型 API Key
          └─ 智能体配置选择其中一个模型 API
```

也就是说，模型管理不是让智能体每次手动填写厂家、模型名和 Key；模型管理先把厂家、模型、Key 维护好，智能体管理只负责选择。

### 1. 模型管理

模型管理负责“外部能力接入”，不直接处理业务逻辑。

当前已落地三类对象：

| 对象 | 表 | 说明 |
| --- | --- | --- |
| 模型供应商 | `ai_model_provider` | 例如智谱、Kimi，记录 `base_url`、认证方式、启用状态 |
| 模型 | `ai_model` | 记录具体可调用模型，例如 `glm-4.7`、`kimi-k2.6` |
| 模型 API Key | `ai_model_api_key` | 一个模型下可以有多个 Key，便于切换免费额度、付费额度或公司 Key |

当前开发库默认初始化：

```text
智谱：glm-4.7、glm-4.5-flash
Kimi：kimi-k2.6
```

当前每个默认模型都初始化了一个对应的模型 API Key：

```text
glm-4.7          -> 智谱 GLM-4.7 Key
glm-4.5-flash   -> 智谱 GLM-4.5-Flash Key
kimi-k2.6       -> Kimi K2.6 Key
```

### 2. 智能体管理

智能体管理负责“给模型赋能”。

即使底层只是一个普通大模型，只要系统给它配置了：

- 系统提示词
- 长期记忆
- 知识库
- 工作流
- 本地工具
- 权限边界

它在产品表现上就可以成为不同层级的 AI 助手或智能体。

当前 `ai_agent` 通过以下字段决定能力形态：

| 字段 | 含义 |
| --- | --- |
| `model_id` | 绑定统一模型管理中的具体模型 |
| `api_key_id` | 指定本次 AI 配置使用的模型 API Key，Key 会自动反推出供应商与模型 |
| `memory_enabled` | 是否启用长期记忆 |
| `knowledge_enabled` | 是否启用知识库 |
| `workflow_enabled` | 是否启用工作流 |
| `tools_enabled` | 是否启用本地工具调用 |

智能体管理页的正确交互是：

1. 先选择供应商。
2. 再从该供应商下筛选出的“模型 API”下拉列表中选择具体模型 Key。
3. 系统自动带出模型名称、供应商编码和 API Key 绑定关系。
4. 智能体管理只配置提示词、能力开关、记忆、知识库、工作流和工具权限，不重复维护模型资源。

### 3. AI 对话

AI 对话是统一使用入口。

后续不应让用户感知太多“这是某平台智能体还是某平台大模型”的技术差异，而应展示为：

```text
普通大模型
增强型 AI
工具型智能体
```

会话记录中可以保留来源标签，用于开发和后台识别，但产品表达应围绕“能力”而不是“厂商宣传名词”。

### 4. 本阶段删除 / 暂缓的内容

`智能体工作台` 第一版用于验证本地工具、记忆、知识库和工作流链路。第二版中，它不再作为主入口继续扩展。

相关能力迁移到：

```text
智能体管理：配置能力
AI 对话：真实使用能力
```

这样更贴近老板真正要测试的“企业 AI 赋能助手”。

## 十、场景层：把 AI 能力组织成客户 / 业务专属团队

模型管理负责“供应商 -> 模型 -> 模型 API Key”的资源维护；智能体管理负责“选择模型并赋能”；场景管理负责把多个 AI 组织到具体业务场景中。

### 1. 为什么需要场景

老板真正关心的不是某一个模型，而是“面对某类企业时，AI 能不能组成一个可复用的解决方案团队”。因此需要一个场景层：

- 面向不同客户或行业建立不同场景。
- 每个场景选择一组可用 AI。
- 场景决定上下文组织方式，而不是替代 AI 能力配置。

### 2. 两种场景模式

| 模式 | 上下文规则 | 适用情况 |
| --- | --- | --- |
| `single` 单聊 | 每个 AI 独立上下文 | 分别测试某个 AI 的能力 |
| `team` 团队 | 场景内 AI 共用上下文 | 模拟多个 AI 在同一会议中协作 |

### 3. 图片生成与场景的关系

图片生成仍然是 AI 助手的工具能力：

- 单聊场景：当前 AI 开启图片生成后，可以在自己的独立上下文中生成图片。
- 团队场景：把带图片生成能力的 AI 加入场景后，用户切换到该 AI，即可基于团队共享上下文生成图片。
- 场景本身不直接生成图片，只负责组织上下文和可用 AI。

这个设计可以避免“一个功能绑死在某一家模型上”，后续任何大模型都可以通过工具赋能获得图片生成、本地文件、知识库、工作流等能力。
