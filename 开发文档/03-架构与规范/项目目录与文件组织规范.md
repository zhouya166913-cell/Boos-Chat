# 项目目录与文件组织规范

本文档用于统一当前四个正式项目的目录划分、文件拆分方式和长期演进规则。

适用范围：

```text
boss-chat-server/     # Java Spring Boot 后端
boss-chat-web/        # Vue 3 + TypeScript + Element Plus 管理系统
boss-chat-app/        # Flutter App
boss-chat-miniapp/    # Vue 3 + uni-app 小程序
```

参考项目 `MinimaxAndroidChat`、`minimax-token-chat-demo`、`tools` 仅保留参考价值，不纳入本规范约束。

---

## 一、组织原则

### 1. 先按业务域划分，再在业务域内部做技术分层

当前项目后续会持续扩展，单纯按 `controller / service / api / views` 这种“技术层目录”继续平铺，短期简单，长期会让同一业务散落在多个地方。

因此统一采用：

```text
先看它属于哪个业务模块
再看它在模块里承担什么职责
```

例如“智能体对话”模块应优先聚合为：

```text
chat/
├─ components/
├─ composables/
├─ styles/
└─ ...
```

而不是让相关文件长期分散在 `views/`、`api/`、`styles/`、`components/` 各处后再靠人记忆拼起来。

### 2. 按职责拆分，不为了拆分而拆分

不是每一个页面都必须机械拆成：

```text
test.vue
test.ts
test.css
```

如果一个页面本身很小、逻辑简单、只服务当前页面，保留为单文件组件是合理的。

只有当出现下面任一情况时，才应拆分：

- 页面同时承载接口请求、复杂状态、多个展示区块
- 单文件已经难以快速定位代码
- 样式明显属于某一模块，可独立维护
- 某一段展示或逻辑未来会被复用
- 新同事看文件时，需要频繁上下跳转才能理解

### 3. 页面文件应尽量“薄”

页面文件主要负责：

- 组合模块能力
- 绑定页面级数据
- 组织最终布局

不应长期承担：

- 大量接口调用细节
- 复杂状态编排
- 重复展示组件
- 与其他页面可复用的样式

换句话说，页面文件应更像“装配层”，而不是“所有东西的最终堆放处”。

### 4. 公共能力和业务能力要分开

统一约定：

- **业务模块**：围绕具体业务能力，例如 `auth`、`chat`、`users`、`agents`
- **共享能力**：跨业务复用，例如 `shared`、`common`、`utils`

禁止把“暂时不知道放哪”的文件一股脑扔进 `common` 或 `utils`。

## 二、四个项目的目标结构

## 1. 后端 `boss-chat-server`

当前后端已经基本采用“按业务域拆包”的方向，后续统一维持：

```text
com.zhiyinhui.bosschat
├─ ai/
│  ├─ bootstrap/
│  ├─ controller/
│  ├─ dto/
│  ├─ entity/
│  ├─ mapper/
│  └─ service/
├─ auth/
│  ├─ controller/
│  ├─ dto/
│  └─ service/
├─ system/
│  ├─ bootstrap/
│  ├─ controller/
│  ├─ dto/
│  ├─ entity/
│  ├─ mapper/
│  └─ service/
├─ common/
│  ├─ config/
│  ├─ exception/
│  └─ ...
└─ BossChatApplication.java
```

### 后端约定

- 新业务模块优先新增独立包，例如未来的 `billing/`、`order/`、`membership/`
- 每个业务模块内部继续保持 `controller / dto / service / entity / mapper`
- `common/` 只放真正跨模块复用的能力
- 已无实际业务价值的示例包不得长期保留在正式主线中
- 数据库变化必须同步 Flyway 迁移脚本

### 什么时候需要进一步拆分

- 单个 Service 已经混合多个业务子域
- DTO 数量明显增加且语义差异大
- 第三方平台接入逻辑变复杂
- 同一模块里出现多种明显不同的业务流程

## 2. 管理系统 `boss-chat-web`

管理系统统一向“模块化前端”演进：

```text
src/
├─ api/                  # 过渡期保留：统一 HTTP 基础能力与少量共享请求
├─ layouts/              # 全局布局
├─ modules/              # 业务模块
│  ├─ auth/
│  │  ├─ composables/
│  │  ├─ styles/
│  │  └─ ...
│  ├─ chat/
│  │  ├─ components/
│  │  ├─ composables/
│  │  ├─ styles/
│  │  └─ ...
│  ├─ users/
│  └─ agents/
├─ router/
├─ stores/
├─ styles/               # 全局基础样式、布局样式、页面共用样式
├─ views/                # 路由入口页面；应尽量保持薄
├─ App.vue
└─ main.ts
```

### 前端文件拆分建议

| 类型 | 适合放置内容 |
| --- | --- |
| `XxxView.vue` | 页面壳、布局组合、少量页面级绑定 |
| `components/` | 某业务模块内可复用的展示块 |
| `composables/` | 页面状态、接口编排、复杂交互逻辑 |
| `styles/` | 某业务模块独有样式 |
| `api/` | 接口请求封装 |
| `stores/` | 跨页面共享状态 |

### Vue 单文件组件何时保留在一个文件里

保留单文件即可：

- 页面很短
- 没有复杂接口编排
- 没有可复用展示区块
- 没有大量独立样式

应拆分：

- 页面逻辑明显超过“页面壳”职责
- `<script setup>` 内已经承载多段独立业务流程
- `<template>` 内出现重复展示区块
- 样式较多且只属于该模块

### 前端样式组织

统一分三层：

```text
styles/
├─ base.css          # reset、字体、根节点等基础样式
├─ layout.css        # 全局布局，例如后台框架、公共标题区
└─ pages.css         # 少量跨页面复用样式

modules/
└─ xxx/
   └─ styles/
      └─ xxx.css     # 某业务模块独有样式
```

不再把登录页、后台布局、聊天页所有样式长期堆进一个 `main.css`。

## 3. Flutter App `boss-chat-app`

App 当前仍处于起步阶段，但正式结构从现在开始固定为：

```text
lib/
├─ app/
│  ├─ routes/
│  └─ theme/
├─ core/
│  ├─ network/
│  ├─ storage/
│  └─ utils/
├─ modules/
│  ├─ auth/
│  │  ├─ models/
│  │  ├─ pages/
│  │  ├─ services/
│  │  └─ widgets/
│  └─ ...
└─ main.dart
```

### Flutter 约定

- `main.dart` 只保留应用启动和最小装配职责
- 页面、模型、接口服务不得长期堆在 `main.dart`
- 业务功能按模块聚合，例如未来的 `chat/`、`profile/`
- 跨模块通用能力放入 `core/`

## 4. 小程序 `boss-chat-miniapp`

小程序也统一采用模块优先的方向：

```text
src/
├─ api/                  # 过渡期保留公共请求
├─ config/
├─ modules/
│  ├─ auth/
│  │  ├─ pages/
│  │  ├─ styles/
│  │  └─ ...
│  └─ ...
├─ pages/                # 路由入口页，可逐步迁移为模块页
├─ shared/
├─ types/
├─ utils/
├─ App.vue
└─ main.ts
```

### 小程序约定

- 页面逻辑与请求逻辑分开
- 页面专属样式可跟随模块放置
- 全局通用能力放 `shared/` 或 `utils/`
- 当某一页面开始出现可复用区块时，及时提取组件

## 三、文件层面的具体规则

## 1. Vue 文件顺序

统一使用：

```vue
<script setup lang="ts">
</script>

<template>
</template>

<style scoped>
</style>
```

如果样式已经迁出到独立 CSS 文件，则组件中可不保留 `<style>`。

## 2. TypeScript 文件规则

- 一类文件只承担一类职责
- 页面状态编排优先放在 `composables`
- 类型定义不要混在大量逻辑中
- 可复用类型应独立导出
- 同一模块内文件命名应让人能从名字推断用途

## 3. CSS 规则

- 一个 CSS 文件尽量对应一种清晰范围
- 不在模块 CSS 中随意污染全局选择器
- 全局基础样式只放真正跨模块通用的内容
- 页面或模块专属样式优先跟随模块放置

## 4. 注释规则

沿用《项目代码规范说明》中的要求：

- 页面、模块、组件要写顶部说明
- 关键方法要说明“做什么”和“为什么”
- 组件要说明输入、输出、依赖
- 后端接口和 DTO 要写清楚中文说明

## 四、当前项目的落地策略

本项目不做“大爆炸式重构”，统一采用分阶段整理：

### 第一阶段：建立规则并先整理管理系统

原因：

- 管理系统目前功能最完整
- 页面、逻辑、样式已经开始增长
- 最适合作为之后 App 和小程序的参照样板

重点：

- 把聊天模块拆成页面壳、状态逻辑、展示组件、模块样式
- 把登录页格式整理到可读状态
- 把全局 CSS 拆成基础、布局、模块几层

### 第二阶段：继续整理后端

重点：

- 清理正式项目中遗留的示例模块
- 继续维持按业务域拆包
- 在未来新增计费、会员、订单等能力时继续沿用同一规则

### 第三阶段：整理 App 与小程序

重点：

- 在真正恢复功能开发前，先把骨架改到目标结构
- 避免功能多起来后再从单文件、平铺目录中返工

---

## 五、判断文件是否需要拆分的快速标准

如果一个文件同时满足下面任意两项，就应优先考虑拆分：

- 超过一个清晰职责
- 包含多个独立业务流程
- 包含较多专属样式
- 新人阅读时需要频繁上下翻找
- 某一部分未来大概率会复用

反过来，如果一个文件：

- 很短
- 很直观
- 只服务一个页面
- 拆出去后反而增加跳转成本

那就继续保留在一起，不做形式主义拆分。

---

## 六、后续新增代码的执行口径

从本文档生效后，后续新增代码默认遵循：

1. 先判断属于哪个业务模块
2. 再判断属于页面、组件、状态逻辑、样式还是共享能力
3. 优先让页面文件保持薄
4. 样式尽量跟随作用域
5. 新模块一开始就按正确目录落位
6. 如果发现历史代码与规范不一致，优先在触碰该模块时顺手收束

这套规则的目标不是让目录“看起来高级”，而是让项目越往后开发，越不容易失控。
